# Implementation Plan: Configuration Templates

**Branch**: `004-field-templates` | **Date**: 2026-03-27 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/004-field-templates/spec.md`

## Summary

Allow users to save, load, rename, delete, export, and import field layout templates.
Templates are persisted in `localStorage` (survive page reloads), exportable as `.json`
files or clipboard text, and importable from files or pasted text. All logic is
client-side only; zero server changes. New entities: `Template` (localStorage record)
and `TemplateFile` (serialized wire format with `schemaVersion: 1`). Field IDs are
regenerated on import to prevent React key collisions.

## Technical Context

**Language/Version**: TypeScript 5.7, React 18.3

`Clipboard API`, browser `Blob + URL.createObjectURL` (no new npm packages)

**Scale/Scope**: Single user, single browser; expected < 50 saved templates

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| Principle III: localStorage persistence | Templates are user preferences that must survive page reloads — the core feature value. Without persistence, templates are useless (gone on every reload). | "Session-only templates" — templates would reset on every page load, defeating the entire purpose. In-URL or server-side storage would violate VIII (no auth) and I (no client-side server persistence). `localStorage` is the correct scope for single-user browser preferences. |

**Note**: This violation requires a MINOR constitution amendment (1.0.2 → 1.1.0) to
carve out an explicit exception: Principle III covers *document editing state* (ephemeral
by design); *user configuration preferences* (localStorage) are a separate category that
is permitted with justification. Amend after implementation is complete.

## Key Implementation Notes

### `useTemplateStore` initialisation

```typescript
const STORAGE_KEY = 'pdf-form-editor:templates';

function loadFromStorage(): Template[] {
  try {
    const raw = localStorage.getItem(STORAGE_KEY);
    if (!raw) return [];
    return JSON.parse(raw) as Template[];  // validated as Template[] shape
  } catch {
    return [];
  }
}
```

State is seeded once from `localStorage` on component mount. Every mutation writes back
via `useEffect` watching the `templates` array. `QuotaExceededError` is caught; if thrown,
the state write is rolled back and `storageError` is set.

### `parseTemplateFile` error messages

The function must produce descriptive errors (FR-013):
- JSON syntax error → "Error de sintaxis JSON: [native message]"
- Missing `fields` → "Plantilla inválida: falta el campo 'fields'"
- `fields` not an array → "Plantilla inválida: 'fields' debe ser un array"
- Invalid field at index N → "Campo en posición [N] es inválido: [property] incorrecto"

### `TemplatePanel` replace/append UI flow

The replace/append choice is shown **inline** (not in a modal) as two buttons that appear
below the import section only when (a) import is valid AND (b) canvas has existing fields.
This avoids the `position:fixed` DOM parentage issue (BF-003-02 class).
